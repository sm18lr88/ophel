# Security Audit

Last updated: 2026-03-08

## Scope

Threat model used for this review:

- Treat the repo as potentially hostile.
- Assume outbound network paths, privilege requests, remote code loading, and HTML injection are untrusted until proven otherwise.
- Keep the product functional, but prefer explicit allowlists and safe defaults over convenience.
- All outside dependencies must be installed locally, without external fetching.
- Only allow communication between the user and approved LLM providers.
- WebDAV sync to be kept but gated strictly with no loophole for connection hijacking.

## Codebase Provenance

- **Supported services**: Originally included Doubao (`www.doubao.com`), a ByteDance (TikTok parent company) AI product. Removed as part of threat model enforcement (see SA-030).
- **WebDAV examples**: Config comments reference Jianguoyun (坚果云), a Chinese cloud storage provider, as an example WebDAV target.

## Hardening Implemented

- Added `src/utils/network-security.ts` as the centralized outbound request policy.
- Restricted WebDAV requests to HTTPS public hosts only.
- Blocked local/private network targets for all validated outbound paths.
- Restricted WebDAV methods to `DELETE`, `GET`, `MKCOL`, `PROPFIND`, and `PUT`.
- Restricted forwarded WebDAV headers to `Content-Type` and `Depth`.
- Restricted watermark proxy fetches to approved `googleusercontent.com` hosts.
- Removed remote runtime script loading from the userscript build.
- Removed automatic remote icon fetch from userscript metadata.
- Converted direct runtime npm dependencies to local vendored `file:` dependencies.
- Sanitized HTML before Trusted Types creation and before adapter DOM insertion.
- Changed extension permission requests from blanket host access to specific HTTPS origin requests.
- Replaced all `window.postMessage(..., "*")` calls with origin-restricted `window.postMessage(..., window.location.origin)` across 11 files.
- Added `event.origin === window.location.origin` checks to all `message` event listeners.
- Added `event.source === window` checks where previously missing (policy-retry-manager, tab-manager) with userscript sandbox compatibility fallback.
- Added strict payload shape validation to network-monitor init messages (requires `urlPatterns` array and numeric `silenceThreshold`).
- Added type-checked `data` object validation before processing in all message handlers.
- Added action allowlist validation to iframe-scroll-main scroll request handler.
- Added boolean type check to scroll-lock-main toggle handler.
- Added `validateLlmProviderUrl` with strict LLM provider domain allowlist (`gemini.google.com`, `business.gemini.google`, `aistudio.google.com`, `chatgpt.com`, `chat.openai.com`, `grok.com`, `claude.ai`, `*.googleusercontent.com`, `*.claude.ai`).
- Locked `platform.fetch()` (both extension and userscript) to LLM provider domains only via `validateLlmProviderUrl`.
- Locked `validateOpenTabUrl` to LLM provider domains or browser settings pages only.
- Locked `open-global-url` keyboard shortcut to LLM provider domains with fallback to `gemini.google.com`.
- Replaced userscript `@connect *` with explicit LLM provider domain list. WebDAV to non-listed hosts triggers Tampermonkey confirmation prompt.
- Restricted `web_accessible_resources` from `<all_urls>` to the 7 LLM provider page patterns.
- Vendored all 22 devDependencies into `vendor/deps/` with `file:` references in `package.json`.
- Added `.npmrc` with `prefer-offline=true`, `verify-store-integrity=true`, and `side-effects-cache=false`.
- WebDAV sync retained as user-gated feature: requires explicit permission grant per-server, HTTPS-only, method/header allowlists, public-host validation.

WebDAV sync targets are NOT in this allowlist. They are gated separately:

- Extension: requires explicit `chrome.permissions` grant for the specific WebDAV origin
- Userscript: triggers Tampermonkey `@connect` confirmation prompt for non-listed domains
- Both: validated via `validatePublicHttpsUrl` (HTTPS-only, no private IPs, no embedded credentials)

## Dependency Audit Procedure

Run `pnpm audit:deps` after any lockfile change to check for new advisories. This command queries the npm registry advisory database and requires network access.

All direct dependencies (both runtime and dev) are vendored in `vendor/deps/` as `file:` references and do not fetch from the registry during install. Transitive dependencies of vendored packages still resolve from the pnpm store, which is populated on first `pnpm install`. The lockfile (`pnpm-lock.yaml`) pins exact versions for reproducibility. Only vendored runtime packages are bundled into the final extension/userscript; dev tooling transitives do not affect the shipped build output.
